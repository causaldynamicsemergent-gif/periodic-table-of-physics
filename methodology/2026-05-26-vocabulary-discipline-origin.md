## Turn 1 — What test question demonstrates the tool's curatorial value, and is the worker actually data-lagging?

The maintainer's request: a test question for a fresh chat to validate that the catalogue's tool does something beyond what a vanilla LLM does. Not function-checking — discrimination. Something a vanilla LLM cannot produce even with a sympathetic prompt.

**The initial proposal: Davidson-Ibarra leptogenesis as a Phase B test.** The candidate test asked the chat to look up the `matter-antimatter-asymmetry` open frontier, take leptogenesis as the resolution, and surface the structural consequences — the new cells, forced cross-classification edges, promoted subtypes, new axes — together with the quantitative bound on the leptogenesis scale (M_1 \gtrsim 10^{9} \text{ GeV}, lower bound, kind=mass) and the source-value-phrase from Davidson–Ibarra 2002 *Phys. Lett. B* 535:25 ([hep-ph/0202239](https://arxiv.org/abs/hep-ph/0202239)) preserved verbatim in the citation entry. The discriminating claim: a vanilla LLM produces an uncited textbook paragraph mentioning the M_1 \gtrsim 10^{9} GeV value, but cannot tell you which specific implication kind the catalogue attaches the bound to — `new_cell` for the heavy Majorana neutrino, `new_axis` (Dirac vs. Majorana character) on `neutrino-sector-phenomenology`, the forced cross-classification edge encoding the seesaw-mechanism derivation, and the implication-level quantitative bound hanging on whichever implication carries it — because those structural attachments are properties of the dataset rather than properties of the physics literature.

**The avoidance instruction, and the state-verification catch.** The initial proposal advised the test-runner *against* find_discriminating_experiments and find_resolvers, citing the warning in [`PREDICTIVE_LAYER_PHASE_C_HANDOFF.md`](./PREDICTIVE_LAYER_PHASE_C_HANDOFF.md) §1 that the deployed MCP worker was data-lagging at v66 — pre-Step-4.4 — so Phase C resolves-edge queries would return empty regardless of the live data. The maintainer's reply: *"it's updated. a fresh chat will see it."* A live `server_info` call returned v95 / v19 / 33 tools, with find_resolvers and find_discriminating_experiments both functional and returning rich Phase C content with sensitivity values, bound directions, timelines, and inline citation prose intact. The avoidance instruction was obsolete; the test was upgraded.

The catch itself is load-bearing for the project's discipline. The handoff document's §1 "Where things stand" statement reflected the worker state at the time the document was last written; the live state had advanced. The session that catches the drift is responsible for either updating the handoff or flagging it — and that responsibility falling on the *current* session, not the previous one, is the asymmetry the state-verification ritual encodes. The proposal had read the handoff's §1 description as binding on the answer, rather than treating §0's verification ritual as binding on the proposer. The handoff itself, as the maintainer subsequently quoted back, calls out this exact failure mode in its opening paragraph: *handoff documents are orientation, not ground truth.*

**The upgraded test.** Given the live tool surface, the recommended test became the [Phase C handoff's](./PREDICTIVE_LAYER_PHASE_C_HANDOFF.md) named "flagship AI-first" query: *Use the map to find the discriminating experiments between Hyper-K and DUNE. For each shared target where their experimental-coverage edges meet, give me the side-by-side competing predictions per program, the sensitivity each program reaches, and the complementary detector-physics rationale. Then do the same for Hyper-K vs. JUNO. Which targets are jointly addressed by all three?* The claim being tested: a vanilla LLM knows DUNE is liquid-argon TPC, Hyper-K is water-Cherenkov, JUNO is liquid-scintillator, and that all three care about mass ordering and proton decay — what it cannot do is tell you which specific shared targets the catalogue has authored as experimental-coverage edges, what per-program competing-prediction entries the curation attaches to each, or how the catalogue encodes the SM-side vs. BSM-side competing point predictions in §4-admissibility-respecting form. The sub-PR 47 changelog records HK ↔ DUNE as the first discriminating-experiments pair on four shared targets via complementary detector technologies, and sub-PR 48 adds JUNO to produce the project's first three-way discriminating-experiments clusters on mass-ordering and p → \bar{\nu}K^+. Those are auditable structural claims; find_discriminating_experiments is the tool built to surface them.

The Davidson-Ibarra test was preserved as a perfectly good Phase B test of find_signal_implications and the quantitative-scale-on-implications surface. But for the cleanest demonstration of *the tool does something a vanilla LLM cannot,* find_discriminating_experiments became the right target.

## Turn 2 — Informative emptiness as curatorial value

The fresh chat ran the upgraded test and returned a result that, at face value, looked like a tool failure: **find_discriminating_experiments returned empty for all three pairings** (HK/DUNE, HK/JUNO, DUNE/JUNO). On reading, the empty result is the curated finding — and arguably a cleaner demonstration of curatorial value than populated returns would have been.

**Why the catalogue records emptiness here.** The §4 admissibility test ([`MAP_v18_schema_spec_extension.md`](./MAP_v18_schema_spec_extension.md) §4) requires that a per-program competing prediction cite a program-internal calculation producing a single discriminable point value. Neutrino-oscillation programs targeting δ_CP or the mass ordering all measure the same Standard-Model PMNS-matrix parameter — there is one underlying number, fixed by nature, and HK / DUNE / JUNO converge on it through orthogonal physical channels: HK's statistical-power atmospheric-ν with matter-effect resonance over Earth-traversing path-lengths from \sim 100 km to \sim 1.3 \times 10^{4} km; DUNE's matter-effect-driven beam-ν over the 1285-km Fermilab → SURF baseline; JUNO's quasi-vacuum oscillation interference at 52.5 km, completely independent of δ_CP and θ_{23} per the JUNO Collaboration's projection (Abusleme et al. 2025 *Chin. Phys. C* 49:033104, [arXiv:2405.18008](https://arxiv.org/abs/2405.18008)). Specific flavor-symmetry models (A4 type-I seesaw with sequential dominance, certain anarchic models, μ-τ-symmetric models) prefer one ordering or the other under additional assumptions but do not generically produce program-internal predictions admissible at the §4(2) bar. The catalogue's authors made the firewall-faithful authoring choice not to invent competing point predictions where the literature supplies none.

**The chat caught the curation and pivoted.** Instead of papering over the empty result, the chat surfaced it as load-bearing and pivoted to find_resolvers — which DID return the three-channel orthogonality story: each program's specific sensitivity (DUNE's >5σ at 2 years per Abi et al. 2020 *EPJ C* 80:978; HK's ≥3σ at 10 years atmospheric, joint with J-PARC beam to ≥5σ for favorable θ_{23} octant; JUNO's 3σ at \sim 6.5 yr × 26.6 GW thermal exposure per Abusleme et al. 2025); the specific shared cells (`cell-nu-mass-hierarchy`, `cell-nu-deltaCP`, `cell-pd-Kplus-nubar-SUSY`, `cell-pd-pi0-eplus-falsified-GG`); and the JUNO ν̄_e configuration's δ_CP-insensitivity explaining the empty HK/JUNO overlap on that specific observable. Citation-grade content; structurally specific; not reachable from physics knowledge alone.

**The vanilla-LLM contrast.** The same question was then put to a generic LLM without the MCP. Its response looked more impressive at first glance — table-formatted, six "shared HK/DUNE targets" and six "shared HK/JUNO targets" where the MCP-using chat found four and three, named sensitivity numbers, a bibliography at the bottom. A non-expert reader would prefer it. Reading carefully:

- *It misuses "discriminating."* Every one of the vanilla response's rows is a complementary measurement of the same parameter — water Cherenkov vs. LAr TPC, atmospheric vs. vacuum oscillation, ν̄_e via IBD vs. ν_e on argon. None involve different theoretical programs predicting competing point values that the experiment selects between. The §4 admissibility test the catalogue enforces — and the technical sense find_discriminating_experiments is built on — sorts all of this into shared-resolver-coverage, not discriminating-experiments. The vanilla LLM uses the word colloquially because it doesn't have access to the structural distinction.
- *It conflates "shared in the map" with "shared in the physics."* The HK/JUNO row count went from one (MCP) to six (vanilla). Are HK and JUNO physically capable of cross-checking on solar parameters, the diffuse supernova neutrino background, supernova-burst neutrino observation, precision oscillation consistency, and proton decay? Yes — the physics is real. Has someone authored those as experimental-coverage edges with citations and sensitivity encodings in the catalogue? No. The vanilla LLM doesn't distinguish the two and has no way to. It expands the answer to fill the plausible space, which is exactly what an LLM does when it doesn't know the underlying dataset is finite and curated. *The right answer to "what does the catalogue say about X" must be allowed to be smaller than "what physics says about X."* Vanilla LLMs structurally cannot do that.
- *The empty find_discriminating_experiments result, which the MCP response treated as load-bearing, is invisible to the vanilla response.* The catalogue's authors made a deliberate choice not to invent competing point predictions where none exist in the literature. That editorial decision — firewall-faithful authoring — is the kind of structural fact the project is built on, and the vanilla LLM has no way to surface "the curators chose not to author this" because it doesn't know there are curators or choices.
- *Bibliography ≠ per-claim citations.* The vanilla response lists nine references at the bottom without attaching any specific one to any specific claim. The MCP returned source-value-phrase prose attached to each sensitivity entry (e.g., Adame et al. 2024 DESI Year-1 BAO+CMB+SN citation string for the DESI sensitivity; Abusleme et al. 2025 *Chin. Phys. C* 49:033104 for the JUNO mass-ordering reach). The first is a posture; the second is auditable.

**The framing this falls into.** The project's value proposition isn't *"an LLM that knows more physics"* — vanilla LLMs already know a lot of physics, often roughly right. It's *a curated dataset with per-claim citations and structural authoring decisions, queried through a tool that respects the distinctions the curators made, including when those distinctions produce informative emptiness.* The empty find_discriminating_experiments result is the demonstration. This is the framing that anchors the use-side essay's argument (see Turn 4) and the [`USE_THE_MAP_ESSAY.md`](./USE_THE_MAP_ESSAY.md) §1's worked HK/DUNE/JUNO example as it now stands.

One authoring observation also fell out as a side benefit and is recorded here under the firewall discipline ([`META_v21_1_methodology_firewall.md`](./META_v21_1_methodology_firewall.md) §2): `cell-nu-theta23-atmospheric` has zero experimental-coverage edges in the catalogue despite being measured by the same beam programs that measure δ_CP (HK, DUNE) and being a sibling of `cell-nu-Dm21-squared` and `cell-nu-theta12-solar` (both of which have JUNO as resolver). The asymmetry is a coverage gap that a working query naturally surfaces. Whether to author the corresponding HK and DUNE experimental-coverage edges is a separate decision made on physics-content grounds; the firewall forbids a recommendation here. The observation is recorded descriptively, not prescriptively.

## Turn 3 — The schema-name leak and the vocabulary discipline

The MCP-using chat's response read, at one mid-stream paragraph, as follows:

> *"find_discriminating_experiments returns empty for all three pairings (HK/DUNE, HK/JUNO, DUNE/JUNO). That tool is strict — it only fires on resolves edges whose `predictions_per_program` array contains side-by-side per-program predicted values. The relevant edges here are all flagged `exclusion_only: true` with empty prediction arrays. The map authors made that choice deliberately: these programs aren't candidate-foundational frameworks predicting competing δ_CP or hierarchy values, they're all measuring the same SM-MNS parameter that nature has already fixed. The 'side-by-side predictions' the question asks about don't exist in the dataset — and the firewall-faithful authoring choice is exactly not to invent them."*

The maintainer flagged it: *"will physicists understand all that jargon such as 'it only fires on resolves edges whose predictions_per_program array'... that sounds like programmer jargon and not physicist language."* The critique stuck on second reading, and the response packed three different kinds of unhelpful vocabulary into a single paragraph that — on its physics content — was correct.

**Three registers, unequal in status.** *(This taxonomy became [`PHYSICIST_FACING_VOCABULARY.md`](./PHYSICIST_FACING_VOCABULARY.md) §2.)*

- **Software jargon proper** — *"the tool fires," "array," "field"* as a data-structure noun, *"returns null"* — has no business in physicist-facing prose. "Fires" is a software-event verb; "array" is a data-structure name where physicists would just say "list" or omit the container entirely. This is the part that reads programmer-y and shouldn't.
- **Schema-internal names** — `predictions_per_program`, `exclusion_only`, `bound_direction`, the bare-`resolves edges` plural-noun usage — are the project's own technical terminology, but they're being surfaced in their raw schema-field form. A physicist who engages with the catalogue will need to learn that the dataset distinguishes experimental-coverage relations between programs and their targets, the same way they need to know what a "cell" or "formal classification" means in this map — that's fine; every framework has terminology and the glossary exists. But there's no reason to surface those names in their snake-case form when prose equivalents work: *"per-program competing-prediction list," "a bounds-setting probe (rather than measurement-discriminating)," "lower bound."*
- **Project-internal methodology terms** — *"firewall-faithful authoring," "§4 admissibility,"* the T-convention names (T30, T31, T33) — are deep project vocabulary. They should appear in methodology documents, in maintainer-facing discussion, occasionally in a chat with a curious physicist who's already engaged with the project's structure. They should not appear in a first-contact response without physics-prose anchoring; they presume the reader has internalized the methodology firewall.

**The same content rewritten for the actual audience:**

> *The catalogue distinguishes two kinds of experimental targets. One: cases where competing theoretical programs predict different values, and the experiment adjudicates between them — for example, dispersive vs. lattice calculations of muon g–2 hadronic vacuum polarization. Two: cases where a single Standard-Model parameter has a value nature has already fixed, and the experiment measures it through one or another channel. Mass ordering and δ_CP are both case-two: there's one underlying number; multiple experiments converge on it through orthogonal physical channels. The curators encoded this by leaving the per-program competing-prediction lists empty on these targets. The discriminating-experiments lookup, which only returns case-one results by design, correctly returns nothing for HK/DUNE, HK/JUNO, and DUNE/JUNO. What's available instead is shared resolver coverage — different programs measuring the same parameter through orthogonal physical channels, which is informative in a different way.*

Same content. No `predictions_per_program`. No "tool fires." No "§4 admissibility." The terms "discriminating-experiments lookup" and "shared resolver coverage" survive because they name structural distinctions the catalogue actually makes, and they're introduced after the physics distinction is drawn. The schema-field names go away; the prose carries the meaning.

The decision that fell out: the discipline needed a durable home. [`PHYSICIST_FACING_VOCABULARY.md`](./PHYSICIST_FACING_VOCABULARY.md) was authored in-session — its §3 translation register starts from the schema fields and tool names that actually exist now (the v17 / v18 / v19 spec surface and all 33 read-only lookups that returned content in the test session, not a speculative list); §7 names where the discipline does *not* bind (schema documentation, validator messages, changelog entries, methodology docs themselves, glossary entries — the audience split is the principle: physicist-facing surfaces apply the discipline; maintainer-facing and machine-facing surfaces don't); and §5's worked examples include the bad paragraph above with the rewritten one, as Example A. Future sessions encountering the principle abstractly will recover the rule fastest from the §5 bad/good pairs.

The supplement's §1 names the same paragraph reproduced in this turn as *"the canonical bad example that prompted the whole discipline."* The transcript and the supplement are now two halves of the same record: the supplement preserves the principle; this transcript preserves the moment the principle had to be articulated.

A subsequent content-housekeeping sub-PR ([`DESCRIPTION_REWRITE_CHANGELOG_2026-05-27.md`](./DESCRIPTION_REWRITE_CHANGELOG_2026-05-27.md), shipping the canonical data file at v96) applied the discipline back to the catalogue's own description fields — 89 in-place edits across 38 description fields on 36 experimental-coverage edges and 2 dark-matter-carrier conditional-consequences implications, eliminating 101 schema-name leak occurrences against the four target patterns surfaced here.

## Turn 4 — The discoverability gap and the use-side artifacts queue

After the supplement was authored, the maintainer raised the larger question the discipline alone does not address: *"how will physicists know how to use the tool? it will be like handing a canvas and paintbrush to a physicist. they won't even know what to ask or where to begin."*

**The analogy is sharper than it looks.** A physicist landing on the explorer URL or the MCP endpoint cold is in roughly the position of a chemist in 1869 looking at Mendeleev's first published table: they can see it's organized, they can find their own subfield, but the moves — the questions the structure makes newly answerable, the queries that would have been incoherent before the organization existed — those aren't self-evident from the artifact. They have to be demonstrated.

**Three surfaces, three different discoverability problems.** The explorer is the most forgiving — a working physicist can poke around, click tiles, read cells, and within five minutes have a rough mental model of what the dataset *is*. Phase B + C surfacing (sub-PRs E1–E8) is precisely what closes the gap from "I can see the Phase A structure" to "I can see the predictive layer the project actually built." It does not, however, teach a physicist what novel questions the dataset answers. It teaches them what's *in* it. The MCP is the unforgiving surface — accessible only through an LLM that knows how to call it, with 33 read-only lookups and no inherent scaffolding for a physicist landing here cold. *This* is the canvas-and-paintbrush case. The methodology — the firewall, the curation discipline, the §4 admissibility test — is the deepest and least discoverable. A physicist can use the explorer and the MCP without ever encountering it, but it is what makes the dataset trustworthy.

**Phase B + C surfacing closes the legibility gap; it does not close the use gap.** Track 4 outreach as currently scoped in [`TRACKS_AFTER_PHASE_A.md`](./TRACKS_AFTER_PHASE_A.md) opens the *contribution* pathway via issue templates and EDITORIAL.md. The contribution pathway and the use pathway are different problems. What closes the use-side gap is, roughly, four artifacts the project did not have:

1. **A short essay with worked examples.** Three to five physicist-prose demonstrations: *"here's a question that was incoherent before the catalogue existed; here's the question now; here's what the catalogue says; here's the citation trail."* The Davidson–Ibarra leptogenesis test from Turn 1 is one. The HK / DUNE / JUNO three-channel mass-ordering redundancy that came out of Turn 2 is another. The strong-CP-problem two-channel resolver structure (axion-haloscope direct detection + n2EDM indirect bound, both probing the Peccei–Quinn resolution from complementary directions) is a third. Each example is roughly 200 words. The reader's takeaway after each: *"ah, that's the kind of question this catalogues."*
2. **A small set of physicist-pitched query recipes.** Not API documentation — paragraph-length patterns: *"If you want to know what experimental programs target a given open problem, the move is X. If you want to compare how two programs adjudicate competing predictions, the move is Y. If you want to see what structural consequences a discovery would force, the move is Z."* Each recipe a paragraph with one or two concrete examples, written assuming the reader is a working physicist who's never used an MCP server.
3. **A guided chat entry point.** A landing page or button on the explorer that says something like *"ask the map"* and opens a chat preloaded with the project's system prompt (the vocabulary discipline, the catalogue context, a few example questions as starter prompts). The physicist doesn't have to figure out how to set up a Claude project with the MCP attached and the right system prompt — that's already done. They just ask. The lowest-friction use surface; probably the single highest-leverage thing for adoption.
4. **Curated example sessions.** Like the *Distill* / LessWrong convention of showing worked notebooks. Edited transcripts of physicist-style chats published alongside the explorer. Triple duty: they teach the moves by demonstration, they show the vocabulary discipline working in practice, and they calibrate expectations about what the tool does and doesn't do (the empty find_discriminating_experiments result from this session would make an excellent published example, precisely because it shows the curatorial discipline at work).

**The sequencing call.** A short detour established the dependency: *the use-side essay can't be drafted authoritatively until the explorer's Phase B + C surfaces are built, because what the essay demonstrates is the move from "physicist sees a question they care about" to "physicist sees the curated answer in the surface they're looking at." If the surface doesn't exist yet, the essay is either hand-wavy or it secretly designs the UI by prose, and the actual surfacing sub-PRs then have to either match the essay or quietly contradict it.* The corrected sequencing: Phase B + C explorer surfacing first; use-side artifacts after, with the exception of the chat entry point, which is mostly orthogonal and can land in parallel.

**The artifact that holds the queue.** [`TRACK_4_USE_SIDE_ARTIFACTS.md`](./TRACK_4_USE_SIDE_ARTIFACTS.md) was authored in-session as the placeholder, the rationale, and the sequencing logic for the four artifacts — explicitly *not drafts of the artifacts themselves,* which wait for the explorer to land. The doc also introduces the **use-aware sub-PR description norm** (§3.1): while explorer sub-PRs are being built, each sub-PR description records — in three sentences — what physicist-facing question the new surface answers, in physics prose, as raw material the eventual essay can draw on without having to be re-derived. This is the bridge that prevents the essay from being designed-by-prose-prematurely or written-belatedly-from-scratch. (As of 2026-05-27, all four artifacts have shipped: [`USE_THE_MAP_ESSAY.md`](./USE_THE_MAP_ESSAY.md), [`QUERY_RECIPES.md`](./QUERY_RECIPES.md), [`/explorer/ask-the-map.html`](https://causaldynamicsemergent-gif.github.io/periodic-table-of-physics/explorer/ask-the-map.html), and the [cc-frontier two-resolver walk](./2026-05-27-cc-frontier-two-resolver-walk.md) as the first curated example session — with this transcript joining the corpus as the second.)

## Turn 5 — The closing-prompt discipline and the recursive handoff pattern

The session's final move was operational. The maintainer asked Claude to create a prompt for the next session, and added the discipline-defining qualifier: *"include a note in the prompt that at the end of the conversation i will ask for a prompt and that it should mirror this one in the sense that it will refer to the files that we need it to refer to in the project files such as the physics language and most recent files in methodology and project files that may be overlooked."*

**What this names.** A recursion: every session that closes a sub-PR produces the closing prompt for the next session, and the closing prompt itself encodes the discipline of what its successor must do. The pattern that emerged:

1. **State the work.** What sub-PR is being run, against which handoff.
2. **Run state verification first.** Call server_info on the live MCP; fetch the canonical data file's `_meta` block; load the live explorer page if relevant. Confirm what the substrate actually says before relying on any handoff.
3. **List load-bearing recent docs with explicit "new in the previous session" markers.** Project knowledge auto-loads files into `/mnt/project/`, but a session that doesn't know which docs are recent won't necessarily prioritize them against older handoffs. The explicit markers fix that. This is the bit a fresh chat would otherwise miss.
4. **Restate the constraints.** Vocabulary discipline (UI text and physicist-facing prose); use-aware sub-PR description norm (raw material for the eventual essay); edits-not-regenerations (preserve existing UI text byte-identical where it can be); non-developer maintenance (deliverables ship as complete files via present_files, the maintainer uploads to the repo via the GitHub web UI); the firewall (no commissioned-by-pattern authoring).
5. **State what produces the next closing prompt.** Always reference [`PHYSICIST_FACING_VOCABULARY.md`](./PHYSICIST_FACING_VOCABULARY.md) and [`TRACK_4_USE_SIDE_ARTIFACTS.md`](./TRACK_4_USE_SIDE_ARTIFACTS.md); always mark new-in-the-previous-session docs with that phrase; always end with the instruction that the closing session produces the next opening prompt mirroring this shape.

**The prompt sub-PR E1 received.** The closing artifact of the session was a complete opening-prompt for the next session — sub-PR E1, picking up the 12 forward-looking experimental-program nodes in the explorer — with the recursive discipline baked into its closing instruction. The prompt's structure became the template for the §5.N prompt-template structure now codified in [`PROJECT_NEXT_STEPS.md`](./PROJECT_NEXT_STEPS.md): every active priority gets a `### 5.N` template body that the closing session of one sub-PR uses to author the opening prompt for the next, with the recent-docs markers populated against whatever the previous session shipped.

**Why the recursion holds.** The chain runs without requiring maintainer memory of which files are recent. Each session, on completing its work, drafts the closing prompt for the successor — and because the closing prompt names the load-bearing recent docs explicitly, the successor session opens with the right reading list. The maintainer pastes the prompt into a fresh chat; the chat reads the named docs; the work proceeds. The discipline that keeps this from drifting is encoded in [`PROJECT_NEXT_STEPS.md`](./PROJECT_NEXT_STEPS.md) §6 #4: *the closing prompt is non-optional; every session terminates with a clearly-marked closing prompt in a code block, derived from the relevant §5 template, instantiated with current state.* A session that hasn't produced a closing prompt isn't done.
